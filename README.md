# Elyrion — PoC агентной системы управления инфраструктурой

## 1. Контекст задачи

### Что это за задача

Проект **Elyrion** представляет собой систему ИИ-агентов для управления облачной инфраструктурой.

Система реализует подход **Agent-Driven Infrastructure Management**, при котором управление инфраструктурой осуществляется через ИИ-агентов.

На текущий момент **основной проект уже существует и развернут**.  
В системе реализован **базовый агент управления инфраструктурой**, который позволяет управлять облачными ресурсами.

### Существующий функционал системы

В текущей версии Elyrion реализован **один агент**.

#### Агент управления облачной инфраструктурой

Этот агент отвечает за управление ресурсами облачных провайдеров.

Под капотом агент использует **Terraform**, который применяется как механизм Infrastructure-as-Code.

Функциональность существующего агента:

- управление инфраструктурой облачных провайдеров
- создание и удаление ресурсов
- управление инфраструктурными конфигурациями
- применение инфраструктурных изменений через Terraform

Таким образом, Elyrion уже способен выполнять **инфраструктурные операции через ИИ-агента**.

Однако текущая система **не решает задачу развертывания пользовательских проектов**.

---

## 2. Что будет разработано в рамках курса

В рамках курса будет разработан **второй агент системы — агент развертывания пользовательских проектов**.

Этот агент будет интегрирован в существующую систему Elyrion.

Его задача — автоматизировать развертывание приложений пользователей на базе уже существующей инфраструктуры.

### Новый агент — агент развертывания проектов

Функции нового агента:

- получение информации о проекте пользователя
- анализ GitHub-репозитория
- формирование плана развертывания
- запуск процесса развертывания
- взаимодействие с агентом управления инфраструктурой

## 3. Изменения в существующем проекте

Для реализации нового агента потребуется внести изменения в основной проект Elyrion.

Эти изменения необходимы для того, чтобы система могла поддерживать **мультиагентную архитектуру**.

Планируемые изменения:

- расширение архитектуры системы для поддержки нескольких агентов
- добавление интерфейса взаимодействия между агентами 
- интеграция нового агента развертывания проектов
- добавление механизма передачи задач между агентами

---

## 4. Для кого создаётся решение

Решение ориентировано на:

- разработчиков-энтузиастов
- создателей pet-проектов
- небольшие команды разработки
- инженеров, которые не хотят тратить много времени на DevOps

Для таких пользователей DevOps-часть разработки часто становится дополнительной нагрузкой.

Обычно необходимо:

- настраивать инфраструктуру
- разбираться с Terraform
- поднимать серверы
- конфигурировать сети
- настраивать деплой

Для небольших проектов это требует значительных временных затрат.

---

## 5. Какая боль существует сейчас

Основные проблемы разработчиков:

### 1. Высокий порог входа в DevOps

Даже для небольшого проекта необходимо понимать:

- облачные инфраструктуры
- инфраструктурные инструменты
- контейнеризацию
- сетевые настройки

### 2. Большие временные затраты на развертывание

Настройка инфраструктуры может занимать значительное время и отвлекает от основной задачи — разработки продукта.

### 3. Избыточная сложность для небольших проектов

Многие инструменты DevOps ориентированы на крупные команды и сложные инфраструктуры.

Для pet-проектов это часто является избыточным.

---

## 6. Что именно сделает PoC на демо

PoC продемонстрирует работу **нового агента развертывания проектов** внутри системы Elyrion.

Будет показан следующий пользовательский сценарий.

### Шаг 1 — Добавление проекта

Пользователь добавляет проект в систему:

- указывает ссылку на GitHub-репозиторий
- предоставляет доступ к репозиторию

---

### Шаг 2 — Запрос на развертывание

Пользователь обращается к агенту с запросом на развертывание проекта.

Например:

> "Разверни этот проект"

---

### Шаг 3 — Формирование плана развертывания

Агент анализирует проект и формирует **план действий**.

Пример плана:
 1. Создать виртуальную машину
 2. Подготовить среду выполнения
 3. Склонировать GitHub репозиторий 
 4. Собрать приложение
 5. Запустить сервис
 6. Открыть сетевой доступ

---

### Шаг 4 — Подтверждение пользователя

Пользователь может:

- подтвердить план
- отклонить его

Это позволяет пользователю контролировать инфраструктурные изменения.

---

### Шаг 5 — Развертывание проекта

После подтверждения:

- агент развертывания инициирует выполнение плана
- операции передаются агенту управления инфраструктурой
- инфраструктура создаётся через Terraform

В результате приложение будет развернуто и доступно пользователю.

Если приложение предполагает сетевой доступ, пользователь получит URL сервиса или IP:PORT

---

## 7. Что НЕ делает PoC (Out of Scope)

Для ограничения сложности демонстрационного прототипа некоторые функции не входят в PoC.

### 1. Агент не изменяет код проекта

Агент не будет:

- редактировать код
- исправлять ошибки
- модифицировать проект

Он занимается только **развертыванием инфраструктуры**.

---

### 2. Поддерживается только GitHub

PoC работает только с:

- GitHub репозиториями

Не поддерживаются:

- GitLab
- Bitbucket
- другие источники кода

---

### 3. Нет автоматического обновления развернутых проектов

Если код проекта изменился, система **не обновляет деплой автоматически**.

Пользователь должен заново инициировать развертывание проекта.

---

### 4. Нет полноценного CI/CD

PoC не включает:

- CI/CD пайплайны
- автоматические обновления
- мониторинг
- управление версиями деплоя

Цель PoC — продемонстрировать **концепцию агентного развертывания проектов**.
